# 旅行世界地图 PRD

## 1. 产品概述

### 1.1 产品名称
旅行世界地图

### 1.2 产品定位
一个以“旅行足迹点亮”为核心体验的互动世界地图应用。用户可以在地图上点击真实地理位置创建地点点位，系统需要判断该点击对应的实际地点信息，并为每个地点维护基础介绍内容。

### 1.3 产品目标
- 让用户以可视化方式记录自己的旅行足迹
- 支持在地图上点击真实位置并识别对应地点后创建点位
- 支持为每个地点查看和编辑基础介绍内容
- 在不依赖后端的前提下，提供可持续迭代的前端基础版本

## 2. 版本目标

### 2.1 本期版本
v1

### 2.2 v1 核心目标
- 完成一个基于 `Vue 3 + Vite + TypeScript` 的单页应用
- 提供可交互的世界地图视图
- 支持用户点击地图后进行真实点位判断并创建点位
- 支持点位高亮显示与选中态展示
- 支持打开地点详情面板，查看和编辑基础内容
- 支持通过本地存储保留用户创建和修改的数据

## 3. 用户场景

### 3.1 核心用户
- 想记录旅行足迹的个人用户
- 想以视觉化方式整理旅行计划或回忆的用户

### 3.2 典型使用场景
1. 用户打开地图后，看到预置示例地点和已保存的个人点位
2. 用户点击地图上的某个真实地理位置
3. 系统识别该点击对应的实际地点信息，并创建一个新的地点点位
4. 系统自动高亮该点位，并打开详情面板
5. 用户填写或补充地点简介，保存后该点位永久显示
6. 用户后续点击已有点位，可再次查看、编辑或删除自建内容

## 4. 功能范围

### 4.1 v1 包含范围
- 世界地图展示
- 地图点击后的真实点位判断
- 基于真实地点信息创建点位
- 点位高亮与选中态样式
- 地点详情面板
- 地点名称、简介编辑
- 点亮状态切换
- 用户自建点位删除
- 本地 JSON 预置数据加载
- `localStorage` 持久化用户数据
- 桌面端右侧抽屉与移动端底部抽屉交互

### 4.2 v1 不包含范围
- 后端服务
- 用户登录与多端同步
- 图片上传
- 标签、游记、时间线、评论等扩展内容
- 多人协作

## 5. 交互设计要求

### 5.1 地图交互
- 地图需要具备真实地理点位判断能力，点击地图后要先识别实际地点，再创建点位
- 地图可使用固定投影底图，但底图需要与真实地理坐标或地理数据建立映射关系
- 点击地图陆地区域时，系统应判断点击位置对应的真实地点信息
- 当点击位置无法识别为有效地点时，系统不创建点位，并给出明确提示
- 新点位创建后默认处于高亮状态
- 新点位创建后自动打开详情面板，并进入编辑状态
- 点击已有点位时，打开该点位的详情面板
- 选中点位的视觉强调强于普通高亮点位

### 5.2 点位状态
- 高亮点位应具备明显的视觉识别，例如发光圆点、描边圆环或扩散效果
- 普通高亮点位与当前选中点位需要有明显区分
- 用户应可在详情面板中切换点亮状态

### 5.3 详情面板
- 桌面端显示为右侧抽屉面板
- 移动端显示为底部抽屉面板
- 面板展示字段至少包括：
  - 地点名称
  - 所属国家/地区
  - 地理坐标
  - 地点简介
  - 点亮状态
- 新建状态下允许取消，取消后不保留空点位
- 用户自建点位允许删除
- 面板支持关闭按钮和 `Esc` 键关闭

## 6. 视觉与体验要求

### 6.1 视觉方向
- 整体风格为“旅行探索感”
- 底图风格偏克制，不抢占点位视觉焦点
- 点位高亮使用暖色系，以突出“已去/想去”的积极情绪
- 桌面端优先保证地图的主视觉占比
- 移动端优先保证地图区域可见与抽屉可操作性

### 6.2 可用性要求
- 所有可点击区域满足最小触控尺寸
- 点位需要支持键盘聚焦
- 焦点态需清晰可见
- 抽屉打开和关闭逻辑需要直观明确
- 长文本简介不能破坏面板布局

## 7. 数据设计

### 7.1 地点模型
统一地点模型 `MapPoint` 字段如下：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `string` | 唯一标识 |
| `name` | `string` | 地点名称 |
| `countryName` | `string` | 所属国家或地区名称 |
| `countryCode` | `string` | 国家或地区代码 |
| `cityName` | `string` | 城市名称，可为空 |
| `description` | `string` | 地点简介 |
| `lat` | `number` | 纬度 |
| `lng` | `number` | 经度 |
| `x` | `number` | 渲染用归一化横坐标，范围 `0-1` |
| `y` | `number` | 渲染用归一化纵坐标，范围 `0-1` |
| `isHighlighted` | `boolean` | 是否点亮 |
| `source` | `seed` \| `user` | 数据来源 |
| `createdAt` | `string` | 创建时间 |
| `updatedAt` | `string` | 更新时间 |

### 7.2 坐标规则
- `lat` 和 `lng` 作为地点的真实坐标主数据
- `x` 和 `y` 作为地图渲染层的归一化坐标，可由真实坐标投影或映射得到
- 点击地图时，系统需先将点击位置映射到真实地理坐标或地理区域
- 当识别到国家、地区或城市信息后，再生成对应地点数据

### 7.3 地点判断规则
- 优先识别点击位置所属的国家或地区
- 若具备城市级数据，可进一步匹配最近城市或明确命中的城市点位
- 若点击海洋、无数据区域或无法完成识别，则不创建点位
- 真实地点信息应作为保存数据的一部分，而不是仅保存屏幕坐标

### 7.4 数据来源
- `seed-points.json`：系统内置演示点位，只读
- `localStorage`：保存用户新增和修改后的点位数据
- 地理识别数据：前端内置的国家边界、城市点位或其他静态地理数据资源

### 7.5 数据合并规则
- 运行时合并预置点位和用户本地点位
- 若存在相同 `id`，用户本地数据优先
- 本地存储异常或损坏时，系统自动回退到默认数据

## 8. 页面与模块设计

### 8.1 核心模块
1. 地图视图模块
   - 负责底图渲染
   - 负责点击取点
   - 负责真实地点识别
   - 负责点位展示、高亮和选中态

2. 详情面板模块
   - 负责展示当前点位信息
   - 负责新建、编辑、删除和关闭交互

3. 本地数据模块
   - 负责读取预置 JSON
   - 负责管理本地点位状态
   - 负责 `localStorage` 持久化

4. 地理识别模块
   - 负责地图点击坐标到真实地点的映射
   - 负责国家、地区或城市名称解析
   - 负责无法识别场景的兜底处理

### 8.2 推荐目录结构
- `src/components`：地图和面板组件
- `src/data`：预置地点 JSON
- `src/data/geo`：地理识别所需的静态边界或点位数据
- `src/stores` 或组合式函数：状态管理与持久化逻辑

## 10. 技术方案要求

### 10.1 技术栈
- `Vue 3`
- `Vite`
- `TypeScript`

### 10.2 技术原则
- 第一版仅做单用户本地体验
- 不接入后端服务
- 优先保证结构清晰，便于后续迭代扩展

### 10.3 真实点位判断实现方案
- v1 采用“地图点击坐标 -> 地理坐标换算 -> 地理数据命中 -> 生成地点信息”的前端静态识别链路
- 国家或地区识别作为必须能力，城市级识别作为增强能力
- 真实点位判断必须在前端本地完成，不依赖实时外部接口

### 10.4 地理数据方案
- 国家或地区级识别使用世界边界数据，例如 `GeoJSON` 或简化后的多边形边界数据
- 城市级识别使用静态城市点位数据，字段至少包含城市名、国家代码、纬度、经度
- 地理数据需要预处理，优先采用压缩、简化、按需加载后的静态资源，避免首屏体积过大
- 地理数据文件建议按能力拆分：
  - `countries.geo.json`：国家或地区边界数据
  - `cities.json`：城市点位数据
  - `projection-config.ts`：地图投影和坐标换算配置

### 10.5 点位识别流程
1. 用户点击地图容器
2. 系统根据容器尺寸和地图投影规则，将屏幕坐标换算为地理坐标 `lat/lng`
3. 使用地理坐标在国家边界数据中做点是否落入多边形判断
4. 若命中国家或地区，再根据城市数据做最近城市匹配或区域内城市匹配
5. 生成标准化 `MapPoint` 数据，并写入状态管理层
6. 打开详情面板，让用户补充或修改展示内容

### 10.6 地点识别优先级
- 第一优先级：国家或地区命中
- 第二优先级：城市点位命中或最近城市匹配
- 当国家已识别但城市无法可靠判断时，允许创建国家级地点点位
- 当国家和城市都无法识别时，不创建点位，并提示用户点击有效陆地区域

### 10.7 城市匹配规则
- 若点击位置与某城市点位距离在预设阈值内，则认定命中该城市
- 若多个城市同时满足阈值，优先选择距离最近的城市
- 若没有城市满足阈值，但已识别到国家，则允许仅保存国家级地点信息
- 城市匹配阈值应支持配置，便于后续按缩放等级或地图尺寸调整

### 10.8 地图投影与坐标映射
- v1 需要固定一套明确的世界地图投影方案，保证底图、点位渲染和命中判断使用同一套规则
- 若底图为 `SVG`，需保存其视口尺寸、地图边界和投影换算关系
- `x/y` 仅作为渲染结果缓存，不作为真实地点识别的主依据
- 点位保存时必须同时保留 `lat/lng` 和当前投影下的 `x/y`

### 10.9 性能要求
- 首屏优先加载基础地图和国家边界识别能力
- 城市数据可延后加载，或在地图初始化完成后异步载入
- 地理判断过程需要控制在用户可接受范围内，单次点击反馈应尽量在短时间内完成
- 大体量地理数据应优先做简化和索引，避免每次点击都全量遍历所有数据

### 10.10 降级与容错策略
- 当城市数据未加载完成时，系统仍应支持国家级识别
- 当国家边界数据加载失败时，系统应禁止创建真实点位，并向用户展示识别能力不可用状态
- 当地图投影配置异常时，系统不得生成错误点位，需进入安全失败状态
- 当识别结果存在不确定性时，优先选择“不创建”而不是创建错误地点

### 10.11 状态管理与存储策略
- 预置点位与用户点位使用统一模型管理
- 用户本地新增点位需保存完整识别结果，包括国家、城市、经纬度与简介
- 本地存储中应保留版本号，便于后续数据结构升级时做迁移
- 本地存储读取失败时，系统应自动回退到仅预置点位状态

### 10.12 推荐实现拆分
- `useMapProjection`：负责屏幕坐标与地理坐标转换
- `useGeoLookup`：负责国家边界命中和城市匹配
- `useMapPoints`：负责点位状态、创建、更新、删除与持久化
- `WorldMap` 组件：负责地图展示和交互事件派发
- `PointDrawer` 组件：负责详情展示与编辑

## 11. 验收标准

### 11.1 初始化场景
- 空数据时地图可正常显示
- 用户可在空地图上通过点击有效陆地区域创建首个点位
- 存在预置 JSON 数据时，点位可正常渲染并可点击
- 国家级地理识别数据可正常加载并参与命中判断

### 11.2 核心交互
- 点击地图有效区域后，系统会识别并返回对应真实地点信息
- 识别成功后，会在对应位置生成一个新点位
- 新点位创建后自动高亮并打开详情面板
- 新点位默认带有国家/地区信息和地理坐标
- 用户填写名称和简介并保存后，刷新页面数据仍存在
- 点击已有点位可再次打开详情并编辑内容
- 切换点亮状态后，地图样式同步更新
- 取消新建不会残留空点位
- 删除自建点位后，地图和本地存储中的对应数据同步移除
- 点击海洋或无法识别区域时，不会创建错误点位，并会给出提示
- 城市数据存在时，可返回正确城市或最近匹配城市
- 城市数据不存在或未命中时，仍可正确返回国家级地点结果

### 11.3 响应式与可用性
- 桌面端右侧抽屉不会压缩地图到不可用状态
- 移动端底部抽屉打开后，仍可识别当前地图区域
- 键盘可以聚焦点位和抽屉按钮
- 焦点态和关闭逻辑清晰可用

### 11.4 边界情况
- 地图边缘点击时，点位不会渲染到容器外
- 点击边界区域时，地点判断结果与实际区域归属一致
- 本地存储为空、异常或损坏时，系统可安全回退
- 地理数据缺失或识别失败时，系统能安全降级并提示用户
- 长文本简介不会导致布局错乱
- 城市数据加载较慢时，不影响国家级识别和基础地图浏览

## 14. 默认假设
- “地图任意地点可以点击”在 v1 中解释为“点击地图后需要先完成真实地点判断，再允许创建点位”
- 内容维护方式优先采用本地配置与本地存储组合方案
- 真实点位判断优先保证国家/地区级准确性，城市级识别可根据静态数据能力逐步增强
- v1 默认允许“国家级成功、城市级缺失”的识别结果存在
- 若无额外视觉要求，默认采用“浅色探索风 + 暖色高亮 + 桌面右抽屉 / 移动端底抽屉”方案
