| Client → Store | User interactions (edit/delete buttons) traverse to Pinia store methods, which call authenticated API endpoints |
| Store → API | updateRecord/deleteSingleRecord make authenticated API calls (PATCH/DELETE with session cookie) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-39-01 | Tampering | PopupTripRecord edit form | mitigate | TimelineEditForm validates date range, notes length (≤1000), tags count (≤10) and length (≤20) per Phase 38 contracts; max-length HTML attributes enforce input limits |
| T-39-02 | Tampering | PopupTripRecord delete button | mitigate | ConfirmDialog requires explicit user confirmation before delete; last-record destructive warning prevents accidental data loss |
| T-39-03 | Info Disclosure | Date conflict check | accept | checkDateConflict reveals only date ranges of sibling records locally — no sensitive data exposure beyond what the user already owns |
| T-39-04 | Elevation | Store method calls | mitigate | updateRecord/deleteSingleRecord use authenticated API with userId scoping from session cookie; row-level security enforced by backend per Phase 36 |
</threat_model>
