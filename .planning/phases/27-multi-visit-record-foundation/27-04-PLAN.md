    ```vue
    <script setup lang="ts">
    import { computed, ref } from 'vue'

    interface Props {
      isSubmitting?: boolean
    }
    const props = withDefaults(defineProps<Props>(), { isSubmitting: false })

    const emit = defineEmits<{
      submit: [{ startDate: string | null; endDate: string | null }]
      cancel: []
    }>()

    const startDate = ref<string>('')
    const endDate = ref<string>('')

    const hasStart = computed(() => startDate.value !== '')
    const rangeInvalid = computed(
      () => endDate.value !== '' && startDate.value !== '' && endDate.value < startDate.value,
    )
    const isValid = computed(() => hasStart.value && !rangeInvalid.value)

    const rangeErrorMessage = computed(() =>
      rangeInvalid.value ? '结束日期不能早于开始日期' : null,
    )

    function handleSubmit() {
      if (!isValid.value || props.isSubmitting) return
      emit('submit', {
        startDate: startDate.value || null,
        endDate: endDate.value || null,
      })
    }

    function handleCancel() {
      emit('cancel')
    }
    </script>

    <template>
      <form
        class="trip-date-form grid gap-3 rounded-2xl border border-[#e6d9ec] bg-[linear-gradient(180deg,rgba(255,255,255,0.92),rgba(253,245,255,0.94))] p-4"
        data-region="trip-date-form"
        data-kawaii-surface="form"
        @submit.prevent="handleSubmit"
    ```

  <how-to-verify>
    前提：Plan 01-04 的所有 automated test 已全绿；`pnpm --filter @trip-map/server exec prisma db push` 已在本地数据库生效。

    步骤 1 — 启动全栈：
    - 打开一个终端：`pnpm dev`（同时起 web + server）。
    - 等待 "Local: http://localhost:5173" 出现。

    步骤 2 — 登录或注册账号，确保 auth 已通过。

    步骤 3 — 新增第一次旅行：
    - 在地图上点击一个未点亮的中国地级市或直辖市（例如北京）。
    - 确认 popup 打开。
    - 点击『点亮』按钮。
    - 期望：**popup 内**（不是新弹窗、不是跳转）展开日期表单，含"开始日期"、"结束日期（可选）"两字段。
    - 填写开始日期 `2025-10-01`，结束日期留空。
    - 点击『保存去访』。
    - 期望：表单收起，摘要出现，文案『已去过 1 次』与『最近一次: 2025-10-01』。
    - 地图上该行政区高亮，marker 点亮。

    步骤 4 — 再记一次：
    - 再次点击同一地点（北京）。popup 再次打开并呈现『已去过 1 次』摘要 + 『再记一次去访』按钮。
    - 点击『再记一次去访』。
    - 期望：popup 内展开日期表单（不跳页、不新弹窗）。
    - 填写开始日期 `2025-11-05`，结束日期 `2025-11-10`。
    - 点击『保存去访』。
    - 期望：摘要更新为『已去过 2 次』与『最近一次: 2025-11-05 - 2025-11-10』。
    - 地图上北京仍保持点亮（D-06：多条记录仍算已去过）。

    步骤 5 — 日期校验 UX：
    - 再次点击北京 → 再记一次。
    - 填写开始日期 `2025-12-10`，结束日期 `2025-12-05`（反序）。
    - 期望：错误文案『结束日期不能早于开始日期』显示，『保存去访』按钮置灰不可点。
    - 把结束日期修正为 `2025-12-15`，错误文案消失，按钮重新可点。
    - 点击取消，表单收起，返回摘要（应仍为『已去过 2 次』）。

    步骤 6 — 刷新恢复：
    - 浏览器刷新页面（F5 或 Cmd-R）。
    - 等待 /auth/bootstrap 完成、地图重新加载。
    - 点击北京。
    - 期望：摘要仍显示『已去过 2 次』与『最近一次: 2025-11-05 - 2025-11-10』；两条记录都被恢复（可通过 DevTools Network 查看 bootstrap response 含两条记录，均带 startDate/endDate）。

    步骤 7 — 跨会话恢复（D-09）：
    - 退出账号（logout）。
    - 以同一账号重新登录。
    - 点击北京。
    - 期望：摘要仍为『已去过 2 次』、『最近一次: 2025-11-05 - 2025-11-10』。

    步骤 8 — 取消点亮（D-06）：
    - 在已点亮的北京上，点击 header 的『已点亮』按钮。
    - 期望：北京 marker 消失，高亮消失；再点击北京（未点亮状态）时，popup 显示的是"未点亮"的新记录表单（不是『已去过 2 次』摘要）。
    - 这证明 unilluminate 按地点级清理了两条记录（D-06）。

    步骤 9 — 旧记录『日期未知』（D-08）：
    - （仅当本地或账号中已有 Phase 27 之前的旧记录）点击一个旧记录对应的地点。
    - 期望：摘要显示『已去过 1 次』与『最近一次: 日期未知』（不是伪造的日期）。
    - 如果没有旧记录可测，跳过本步。

    通过标准：以上 8 或 9 步均符合预期。若有任一步不符，在 resume 信号中描述问题点。

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| 浏览器 DOM 事件 (input change) → TripDateForm.vue ref | Vue v-model 直接绑定 string；攻击面仅限 `<input type="date">`，浏览器内置阻止非日期字符 |
| TripDateForm emit → PointSummaryCard handler → store.illuminate → POST /records | UI 层校验（前端 UX 防护），最终拦截由 Plan 02 DTO `@Matches(YYYY-MM-DD)` + service 层 `endDate >= startDate` 提供 |
| store.tripsByPlaceId → LeafletMapStage computed → props → PointSummaryCard 摘要 | 读路径，无用户输入经过 |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-27-04-01 | Tampering | 用户在 DevTools 手动修改 startDate 值为非 YYYY-MM-DD | mitigate | `<input type="date">` 浏览器原生拒收非法格式；服务端 DTO `@Matches` 二次阻断；前端 TripDateForm.isValid 避免空值与反序提交 |
| T-27-04-02 | Tampering | 用户伪造 `emit('illuminate', { startDate: 'evil' })` | mitigate | 服务端 DTO 校验是唯一权威防线（Plan 02 T-27-02-01）；前端仅提供 UX 校验，不作信任边界 |
| T-27-04-03 | Information Disclosure | MapContextPopup 把他人记录摘要错误渲染 | mitigate | activePointTripCount 从 mapPointsStore.tripsByPlaceId 读，store 经 auth-session 的 boundary 切换清空（已有机制）；本 plan 不改 auth-session |
| T-27-04-04 | Denial of Service | 用户反复点击『再记一次』快速提交多次 | mitigate | TripDateForm.isSubmitting 与 store.pendingPlaceIds 配合显示 disabled 状态；但后端不受前端速率限制保护——依赖既有 API 速率限制层（本 plan 不新增保护，accept 风险，单用户速率极低） |
</threat_model>
