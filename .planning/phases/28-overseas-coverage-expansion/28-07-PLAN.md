    1. 修改 `apps/server/scripts/backfill-record-metadata.ts`，把以下三段逐条更新全部从 `update()` 改为 `updateMany()`：
       - `prisma.travelRecord.update(...)`
       - `prisma.smokeRecord.update(...)`
       - `prisma.userTravelRecord.update(...)`

       目标写法固定为：
       - `where: { id: row.id }`
       - `data` 继续覆盖 `datasetVersion`、`displayName`、`regionSystem`、`adminType`、`typeLabel`、`parentLabel`、`subtitle`
       - 若 `result.count === 1`，按原逻辑累计 `matched*Rows`
       - 若 `result.count === 0`，把该行写入新的 skipped summary，并继续处理下一行
       - 若 `result.count > 1`，直接抛错，因为按 `id` 更新不应命中多行

    2. 扩展 `BackfillSummary` 与 CLI JSON 输出，新增以下 3 个字段，字段名固定如下：
       - `skippedTravelRows: Array<{ id: string; placeId: string }>`
       - `skippedSmokeRows: Array<{ id: string; placeId: string }>`
       - `skippedUserTravelRows: Array<{ id: string; placeId: string }>`

       保留现有 `matched*Rows` / `unmatched*Rows`；不要把 zero-count row 误记为 unmatched，因为 canonical lookup 其实命中了，只是目标行在更新时消失。

    3. 抽取一个共享 helper，建议命名为 `applyBackfillUpdate()` 或等价名称，用来统一三张表的 `updateMany()` + `count` 分支，避免 `travelRecord` / `smokeRecord` / `userTravelRecord` 三处再次分叉回 race-prone 实现。

    4. 修改 `apps/server/test/record-metadata-backfill.e2e-spec.ts`：
       - 保留现有 canonical helper happy path 断言
       - 增加一条 unit-style regression，使用 `vi.fn()` 构造最小 Prisma delegate mock 或等价测试夹具，显式模拟 `updateMany()` 返回 `count: 0`
       - 断言 `backfillRecordMetadata()`（或抽出的 helper）不会抛错，并把对应 row 记入 `skippedUserTravelRows`
       - 同时断言写回 payload 仍保持 `datasetVersion === 'canonical-authoritative-2026-04-21'`，明确保留 28-04 的 canonical 语义

    5. 不改 `buildCanonicalMetadataLookup()`、`auth.service.ts`、`records.repository.ts`；本计划只硬化 backfill 更新策略和审计输出。
