<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| auth form submit → API client error classification | 同样是 401，但登录失败与会话失效代表完全不同的安全/UX 语义 |
| `auth-session` store → `map-points` store | 一旦误把 auth submit 失败当作 unauthorized，就会错误清空当前账号 records 边界 |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-23-18 | Information Disclosure | `auth-session` unauthorized recovery | mitigate | 仅真实 session unauthorized 才触发 `handleUnauthorized()` 与 records boundary reset，auth-submit 401 直接回传给 dialog |
| T-23-19 | Denial of Service | auth dialog submit flow | mitigate | auth-submit 失败保留弹层和当前状态，避免错误密码把用户从当前浏览上下文中强制踢回 anonymous |
| T-23-20 | Spoofing | username input contract | mitigate | 前端把注册用户名限制到与后端一致的 trim 后 `2..32`，阻止空白/超长用户名进入身份展示链路 |
</threat_model>
