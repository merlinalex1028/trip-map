<task type="auto">
  <name>Task 1: 用 taze 做 workspace 级 latest 盘点并确认实际升级目标</name>
  <files>package.json, apps/web/package.json, apps/server/package.json</files>
  <action>
1. 在仓库根目录运行 `pnpm dlx taze latest -r --include-locked`，以 taze 的输出作为本次可升级依赖清单的唯一依据。不要手工查 npm 页面，也不要先入为主跳过某些包。

2. 盘点范围限定为当前 workspace 内已存在的 manifests：
   - 根 `package.json`
   - `apps/web/package.json`
   - `apps/server/package.json`
   `packages/contracts/package.json` 当前没有外部依赖，只作为扫描对象存在；若 taze 未报告它有可改动项，就保持该文件不变。

3. 记录 taze 报告中的候选升级，特别留意：
   - 根工具链：`turbo`、`typescript`、`vitest`
   - 前端工具链/运行时：`vue`、`vite`、`@vitejs/plugin-vue`、`vue-tsc`、`pinia`
   - 后端框架/ORM：`@nestjs/*`、`fastify`、`prisma`、`@prisma/client`

4. 不在本任务里做手工版本修改；这一步只确定“taze 将要写哪些文件、哪些依赖会变化”，为下一步实际写入与安装做边界确认。
  </action>
  <verify>
    <automated>pnpm dlx taze latest -r --include-locked</automated>
  </verify>
  <done>taze 输出已覆盖 root/web/server 三个 manifest；候选升级项已明确；没有任何手工修改版本号的动作混入盘点过程。</done>
</task>

<task type="auto">
  <name>Task 2: 执行 taze latest 升级并同步 pnpm lockfile</name>
  <files>package.json, apps/web/package.json, apps/server/package.json, pnpm-lock.yaml</files>
  <action>
1. 从仓库根目录运行 `pnpm dlx taze latest -r --include-locked -w`，直接把 latest 版本范围写回 root/web/server 三个 manifest。

2. 写回后执行 `pnpm install`，让 `pnpm-lock.yaml` 与 manifests 同步；本任务只接受“manifest + lockfile 一起落地”的结果，不保留半完成状态。

3. 保持改动最小：
   - 不改 `workspace:*` 引用
   - 不顺手引入 catalogs / overrides / patches
   - 不为了“看起来更整洁”重排 package.json 其他无关字段

4. 若 `pnpm install` 因某个新版本的 peer/engine/resolution 问题失败，优先做最小的 manifest 级回退，而不是扩散到业务代码：
   - 仅回退报错依赖本身或紧邻的配套包
   - 回退方式优先选“恢复当前 HEAD 中该依赖原始范围”或“重新执行 taze 时用 `-x <pkg>` 排除该包后重写”
   - 回退后重新执行 `pnpm install`

5. 若必须依赖更大范围的配置/代码兼容修改才能完成安装，本 quick 任务到此为止：保留最后一个可安装的 manifests + lockfile 组合，并在 summary 标出阻塞依赖，等待后续单独处理。
  </action>
  <verify>
    <automated>pnpm install --frozen-lockfile</automated>
  </verify>
  <done>root/web/server 的依赖声明已由 taze 写回；`pnpm-lock.yaml` 与 manifests 一致；`pnpm install --frozen-lockfile` 可通过，证明不存在未同步的锁文件差异。</done>
</task>

   - 哪个包被回退
   - 哪个命令原本失败、回退后恢复

5. summary 需要清楚记录最终状态：是“全量 latest 通过”，还是“latest + 少量回退后通过”。
