# Feature Landscape: v7.0 旅行记录编辑与删除

**Domain:** 既有旅行地图产品的新增里程碑特性研究
**Researched:** 2026-04-28
**Confidence:** MEDIUM-HIGH
**Scope lens:** 只覆盖 v7.0 新能力（编辑、元数据、删除），不重复现有时间轴、统计、点亮逻辑。

## 需求类别

1. **记录编辑**
   用户能修改已有旅行记录的日期（开始/结束），而不改变关联地点。
2. **元数据扩展**
   用户能为旅行记录附加备注文本和/或标签，增强记录的个人表达。
3. **单条删除**
   用户能删除同一地点下的某一次旅行记录，而非清除该地点的所有记录。
4. **操作确认**
   编辑和删除操作前有明确的确认机制，防止误操作；无撤销历史。

## Table Stakes

| 类别 | Feature | 用户视角行为 | v7.0 依赖/前置 |
|------|---------|-------------|----------------|
| 记录编辑 | 修改旅行日期 | 用户可以在记录详情中修改开始/结束日期，保存后时间轴和统计同步更新 | 需要 `TravelRecord` 的 `startDate`/`endDate` 字段可更新；时间轴和统计需响应变更 |
| 记录编辑 | 日期校验 | 结束日期不能早于开始日期；不能选择未来日期（可选） | 需要前端表单校验逻辑 |
| 记录编辑 | 编辑入口 | 用户可以从时间轴或地图 popup 进入编辑模式 | 需要在现有 UI 中添加编辑按钮/入口 |
| 元数据扩展 | 添加备注 | 用户可以为旅行记录添加自由文本备注（如"和家人一起的暑假旅行"） | 需要在 `TravelRecord` 中新增 `note` 字段（可选文本） |
| 元数据扩展 | 备注显示 | 备注在时间轴和地图 popup 中可见 | 需要更新时间轴卡片和 popup 组件 |
| 单条删除 | 删除单次记录 | 用户可以选择删除某一条旅行记录，而不影响同地点的其他记录 | 需要 DELETE API 支持 record-level 而非 place-level；前端需确认当前操作粒度 |
| 单条删除 | 删除后状态 | 如果删除后该地点无其他记录，地点应自动取消点亮 | 需要后端检查关联记录数，必要时更新点亮状态 |
| 操作确认 | 编辑确认 | 用户修改日期后，需点击"保存"确认；可取消返回 | 标准表单交互模式 |
| 操作确认 | 删除确认 | 用户点击删除后，弹出确认对话框，明确告知将删除的内容 | 需要确认弹窗组件；Apple HIG / Material Design 均建议删除前确认 |

## Differentiators

| 类别 | Feature | 为什么有价值 | 适合 v7.0 吗 |
|------|---------|-------------|--------------|
| 元数据扩展 | 标签系统 | 用户可以用标签（如"海滩"、"美食"、"徒步"）分类旅行记录，增强回忆和筛选能力 | 可做轻量版（预设标签 + 自定义），但增加 UI 和数据模型复杂度 |
| 元数据扩展 | 标签筛选 | 在时间轴中按标签筛选记录 | 依赖标签系统，可作为后续迭代 |
| 记录编辑 | 批量编辑 | 选择多条记录批量修改日期或添加标签 | 复杂度高，v7.0 不建议 |
| 删除体验 | 软删除 + 回收站 | 删除后保留 30 天可恢复 | 增加数据模型和 UI 复杂度，v7.0 不建议 |
| 删除体验 | 删除动画 | 删除时的过渡动画，增强操作反馈 | 低成本高体验，可考虑 |
| 元数据扩展 | 旅行心情/评分 | 用 emoji 或星级表示旅行体验 | 有趣但非核心，可延后 |
| 元数据扩展 | 旅行伙伴 | 记录同行者信息 | 社交属性，超出单用户范围 |

## Explicit Anti-Features

| Anti-Feature | 为什么本里程碑不要做 | 替代/收敛方案 |
|--------------|----------------------|----------------|
| 编辑历史/撤销栈 | 实现复杂（需要版本快照、冲突解决），且当前产品定位是个人记录工具而非协作文档 | 仅提供确认弹窗，不提供 undo |
| 修改关联地点 | 会触发 placeId / boundaryId 级联更新，影响点亮状态、统计口径、时间轴归类 | 编辑仅限日期和元数据，地点不可变 |
| 富文本备注 | 支持 Markdown 或富文本会增加存储、渲染和迁移复杂度 | 仅支持纯文本备注，限制字符数 |
| 旅行日记系统 | 完整的日记系统（多段落、图片、位置标记）会大幅偏离当前 MVP 范围 | 备注仅作为简短补充，不做正文 |
| 批量删除 | 需要多选 UI、批量确认、批量状态更新 | 逐条删除，每次确认 |
| 自动备份/导出 | 增加同步和文件系统复杂度 | 先不做，后续可考虑 |

## 功能依赖与顺序

```
编辑日期 → 依赖现有 TravelRecord.startDate/endDate 字段可更新
         → 依赖时间轴和统计响应变更

添加备注 → 需要在 TravelRecord 中新增 note 字段
         → 需要 contracts 更新类型定义
         → 需要 Prisma schema 迁移
         → 需要前端表单和显示组件

单条删除 → 需要 DELETE API 支持 record-level
         → 需要后端检查关联记录数并更新点亮状态
         → 需要前端确认弹窗

确认弹窗 → 可复用通用 Modal 组件
         → 编辑和删除都需要
```

## 推荐实现顺序

### Phase 1: 数据模型扩展（contracts + server）

1. 在 `packages/contracts` 中更新 `TravelRecord` 类型，添加 `note?: string` 字段
2. 在 Prisma schema 中添加 `note` 字段（可选文本）
3. 生成并运行 migration
4. 更新 server 的 CRUD API 以支持 `note` 字段

### Phase 2: 编辑功能（前端 + 后端）

1. 创建编辑表单组件（日期选择器 + 备注输入）
2. 添加编辑 API 端点（PATCH /records/:id）
3. 在时间轴卡片和地图 popup 中添加编辑入口
4. 实现日期校验逻辑
5. 编辑后刷新时间轴和统计

### Phase 3: 删除功能（前端 + 后端）

1. 添加删除 API 端点（DELETE /records/:id）
2. 后端实现删除后检查关联记录数，必要时更新点亮状态
3. 创建确认弹窗组件
4. 在时间轴卡片和地图 popup 中添加删除入口
5. 删除后刷新时间轴、统计和地图点亮状态

### Phase 4: 体验打磨

1. 备注在时间轴和 popup 中的显示优化
2. 编辑/删除操作的 loading 和错误状态处理
3. 边界情况处理（删除最后一条记录、编辑后日期冲突等）

## 复杂度评估

| 功能 | 复杂度 | 主要挑战 |
|------|--------|----------|
| 编辑日期 | 中 | 需要确保时间轴和统计同步更新；日期校验 |
| 添加备注 | 低 | 数据模型扩展简单；UI 需要适配现有风格 |
| 单条删除 | 中-高 | 需要处理点亮状态级联更新；确认交互 |
| 确认弹窗 | 低 | 可复用通用 Modal 组件 |

## 关键决策点

| 决策 | 选项 | 建议 | 理由 |
|------|------|------|------|
| 备注字段类型 | 纯文本 vs 富文本 | 纯文本 | 降低复杂度，符合"简短补充"定位 |
| 备注字符限制 | 无限制 vs 500 字符 vs 1000 字符 | 500 字符 | 足够表达，避免滥用 |
| 标签是否纳入 v7.0 | 是 vs 否 | 否（作为 differentiator 延后） | 增加 UI 和数据模型复杂度，先做编辑和删除 |
| 删除后是否保留地点点亮 | 保留 vs 取消 | 取消（如果无其他记录） | 逻辑一致：点亮 = 有旅行记录 |
| 编辑入口位置 | 时间轴卡片 vs popup vs 两者 | 两者 | 用户可能从不同入口进入 |

## 结论

v7.0 的 table stakes 是"编辑日期 + 添加备注 + 单条删除 + 确认弹窗"闭环。标签系统、软删除、批量操作等 differentiators 可以延后。关键挑战在于删除后的点亮状态级联更新和编辑后的时间轴/统计同步。建议按"数据模型 → 编辑 → 删除 → 打磨"顺序实现。
